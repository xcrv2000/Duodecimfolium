## 问题描述

在代码审查中发现，对修饰珠（Modifiers/Beads）的设计产生了**根本性误理解**。

### 正确的设计理念 ✅

修饰珠是**永久的配件/饰品**：
- 玩家获得修饰珠后就永久拥有
- 修饰珠通过"镶嵌"到卡上来改变卡的属性（速度、标签等）
- 镶嵌是一个配置操作（保存在 Deck 中），**不会消耗**修饰珠
- 同一个修饰珠可以在多张卡上同时使用
- 失败或死亡时，修饰珠库存不受影响

### 错误的假设 ❌

重构代码中基于了错误的假设：
- 修饰珠会被"消耗"（在地牢中使用就减少）
- 地牢失败需要"退还"修饰珠
- 地牢进入需要"检查库存充足"

这些假设导致实现了不应该存在的机制。

## 受影响的代码

### 1. Player.ts (src/core/domain/Player.ts)

**当前代码** (❌ 错误注释):
```typescript
/**
 * 修饰珠库存
 * modifierId -> 拥有数量
 * 
 * 这是全局库存（玩家永久拥有）。
 * 在地牢中使用修饰珠时，会从这里减少。      // ❌ 这是错的
 * 入场时检查库存>= 所有卡组镶嵌的修饰珠总数   // ❌ 这是错的
 */
modifiers: Record<string, number>;
```

**应该改为** (✅ 正确注释):
```typescript
/**
 * 修饰珠库存（永久配件库）
 * modifierId -> 拥有数量
 * 
 * 玩家永久拥有的修饰珠。
 * 修饰珠通过在卡组中"镶嵌"来改变该卡的属性，不会被消耗。
 * 一个修饰珠可以在多张卡上同时使用。
 */
modifiers: Record<string, number>;
```

### 2. playerStore.ts (src/stores/playerStore.ts)

**问题代码**:
```typescript
removeModifier: (id, count = 1) => set((state) => ({
  modifiers: { ...state.modifiers, [id]: Math.max(0, (state.modifiers[id] || 0) - count) }
})),
```

**状态**: ❌ 这个 API 不应该用于"消耗"修饰珠

**建议**:
- 🟡 标记为 `@deprecated`，加注释说明只用于数据修复
- 或 🔴 完全删除（如果确定没有其他用途）

### 3. battleStore.ts (src/stores/battleStore.ts)

**问题代码**:
```typescript
collectedLoot: { gold: number, modifiers: string[] };
// ... 通关时
collectedLoot.modifiers.forEach(modId => playerStore.addModifier(modId));
```

**问题分析**:
- collectedLoot 中的 modifiers 表示"掉落品"
- 通关后调用 addModifier 给玩家添加修饰珠
- 这暗示修饰珠是从地牢掉落得到的**消耗品**

**待确认**:
- 地牢是否真的掉落修饰珠？
- 如果掉落，是"获得新类型"还是"增加数量"？
- 目前的实现是"增加数量"，是否符合设计？
