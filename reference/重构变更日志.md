## ✅ 已发现的设计误理解（已修正 - 2026-03-05）

### ✅ 问题：修饰珠不应该被消耗（已完全修正）

**背景**：  
在 Phase 3.1 中，对修饰珠库存的设计存在**根本性误理解**。修饰珠是永久的"饰品"或"配件"，用于改变卡的属性，应该**不会被消耗**。

**受影响的代码**:

1. **[Player.ts](../src/core/domain/Player.ts) 第 14 行注释** (❌ 错误)
   ```typescript
   * 在地牢中使用修饰珠时，会从这里减少。
   * 入场时检查库存>= 所有卡组镶嵌的修饰珠总数
   ```
   应该改为：
   ```typescript
   * 玩家永久拥有的修饰珠库存
   * 修饰珠是将其"镶嵌"到卡上的饰品，不会被消耗
   ```

2. **[playerStore.ts](../src/stores/playerStore.ts) 第 65-67 行** (❌ 应该删除或标记为弃用)
   ```typescript
   removeModifier: (id, count = 1) => set((state) => ({
     modifiers: { ...state.modifiers, [id]: Math.max(0, (state.modifiers[id] || 0) - count) }
   })),
   ```
   这个 API 不应该用于"消耗"修饰珠，只在特殊场景下使用（如修复错误数据）

3. **[battleStore.ts](../src/stores/battleStore.ts) 第 28 行 & nextStage 中的掉落逻辑** (⚠️ 需确认)
   ```typescript
   collectedLoot: { gold: number, modifiers: string[] };
   // ... 通关时
   collectedLoot.modifiers.forEach(modId => playerStore.addModifier(modId));
   ```
   这可能是错的。"掉落修饰珠"应该重新设计为"获得新修饰珠类型"的机制，保证修饰珠不会失去。

4. **[重构变更日志.md](./重构变更日志.md) Phase 3.1** (❌ 描述有误)
   ```markdown
   地牢进入检查：库存数量 >= 本次地牢所有卡组镶嵌珠子总数
   失败时退还修饰珠的逻辑
   ```
   这些描述暗示修饰珠会被消耗，应该删除。

**正确的设计应该是**：
- ✅ 修饰珠是永久配件，拥有就永久拥有
- ✅ 镶嵌修饰珠到卡上是配置操作（Deck 中的 modifierSlots），不消耗珠子
- ✅ 地牢掉落修饰珠 = 获得新的修饰珠类型（玩家库存中数量 +1）
- ✅ 失败时不需要特殊处理（修饰珠丝毫未受影响）

**修正完成** (2026-03-05)：
- ✅ [Player.ts](../src/core/domain/Player.ts) 中的注释已修正（说明为"永久配件"）
- ✅ [playerStore.ts](../src/stores/playerStore.ts) 中 removeModifier 标记为 `@deprecated`
- ✅ [battleStore.ts](../src/stores/battleStore.ts) 中掉落逻辑功能正确（增加修饰珠数量）
- ✅ 不需要额外处理，设计实现已一致

### Change 2.4: 越界速度与边界值处理

**文件**: [src/core/systems/BattleLoop.ts](../src/core/systems/BattleLoop.ts) § recalculateCardSpeed()

**改动**:

#### 下限处理（< 0）:
```typescript
if (speed < 0) speed = 0;
```
- 含义：卡速度不能为负，最小 0（在 tick 0 触发）
- 这符合文档需求："减速不应低于 0"

#### 上限处理（>= 130）:
```typescript
if (speed >= 130) {
  speed = null;  // 标记失效
}
```
- 含义：speed >= 13.0（currentSpeed10 >= 130）超出时间轴范围
- 时间轴范围是 [0, 12.9]，对应 currentSpeed10 ∈ [0, 129]
- 失效卡在 processTick() 中被跳过，永不触发

**UI 显示**（BattleView 中实现）:
- 失效卡显示为半透明或带 ✗ 标记
- 鼠标悬停提示失效原因

- [src/core/systems/BattleLoop.ts](../src/core/systems/BattleLoop.ts)
- [src/core/systems/CardScripts.ts](../src/core/systems/CardScripts.ts) 
- [src/core/domain/Battle.ts](../src/core/domain/Battle.ts)
- [src/features/battle/BattleView.tsx](../src/features/battle/BattleView.tsx)
- [src/data/buffs.json](../src/data/buffs.json)

**核心改进**:

#### 1. 护甲转换为Buff系统
**问题**: 护甲作为BattleUnit的单独字段存在，与buff系统分离
**解决**: 将护甲完全转换为buff系统，统一状态管理

```typescript
// ❌ 旧设计：护甲是单独字段
interface BattleUnit {
  armor: number;  // 单独的护甲字段
  buffs: UnitBuff[];
}

// ✅ 新设计：护甲作为buff
// buffs.json 中定义 armor buff
{
  "id": "armor",
  "name": "护甲",
  "description": "减少受到的伤害 {level} 点",
  "type": "buff",
  "stackRule": "replace",
  "duration": 1
}
```

#### 2. 统一Buff类型字段
**问题**: 不同buff接口的type字段不一致
**解决**: 在BaseBuff中统一type字段为 'buff' | 'debuff'

```typescript
// ✅ 统一后的BaseBuff接口
interface BaseBuff {
  id: string;
  level: number;
  duration: number;
  type: 'buff' | 'debuff';  // 统一的类型字段
}
```

#### 3. 实现描述模板替换
**问题**: buff描述中的{level}占位符未被替换
**解决**: 添加formatBuffDescription函数，支持动态描述

```typescript
// 新增的描述格式化函数
function formatBuffDescription(buffDef: BuffDefinition, level: number): string {
  return buffDef.description.replace(/\{level\}/g, level.toString());
}
```

#### 4. 完善Buff生命周期回调
**问题**: 缺少onTurnStart等重要回调函数
**解决**: 添加startTurn方法，支持回合开始时的buff效果

```typescript
// 新增的回合开始处理
startTurn(): void {
  // 处理所有单位的onTurnStart回调
  this.units.forEach(unit => {
    unit.buffs.forEach(buff => {
      if (buff.onTurnStart) {
        buff.onTurnStart(unit, this);
      }
    });
  });
}
```
